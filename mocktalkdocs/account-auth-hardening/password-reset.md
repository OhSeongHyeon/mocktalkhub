# 비밀번호 재설정 설계

## 목표

- "비밀번호 찾기"가 아니라 "비밀번호 재설정"으로 설계한다.
- 계정 존재 여부를 숨기고, 짧은 수명의 1회용 토큰으로만 재설정한다.
- 재설정 성공 시 기존 refresh 세션을 모두 폐기한다.

## 현재 상태

- 별도 비밀번호 재설정 API가 없다.
- 현재 Access Token은 무상태 JWT이고, Refresh Token은 Redis 기반 회전 구조다.
- 특정 사용자 기준 전체 세션 폐기용 인덱스는 아직 없다.

## 권장 UX 흐름

1. 사용자가 이메일을 입력하고 재설정 요청을 보낸다.
2. 서버는 항상 동일한 성공 메시지를 반환한다.
3. 유효한 로컬 계정이면 재설정 메일을 발송한다.
4. 사용자는 메일 링크에서 새 비밀번호를 입력한다.
5. 서버는 토큰 검증 후 비밀번호를 변경하고 모든 refresh 세션을 폐기한다.
6. 사용자는 다시 로그인한다.

## 응답 원칙

- 요청 단계에서는 계정 존재 여부를 절대 노출하지 않는다.
- 아래 경우에도 응답은 동일해야 한다.
  - 없는 이메일
  - 이미 탈퇴한 계정
  - 소셜 전용 계정
  - rate limit 초과

## 토큰 정책

- 저장은 해시만 사용
- 1회용
- TTL은 `15분` 또는 `30분`
- 새 요청 시 이전 미사용 토큰은 만료 처리
- 재설정 성공 시 즉시 사용 완료 처리

## 비밀번호 정책 권장

- 최소 10자 이상 권장
- 최대 64자 유지
- 이전 비밀번호와 동일한 값 금지
- 너무 흔한 비밀번호 차단 검토
- 서버 검증이 최종 기준이어야 한다

## 세션 무효화 설계

### 필요한 이유

- 현재는 특정 refresh 토큰 단위 폐기만 가능하다.
- 비밀번호 재설정 후 기존 기기 세션을 유지하면 운영 수준 보안 요구에 못 미친다.

### 권장 1차 구현

- Redis에 사용자별 refresh 세션 인덱스를 추가한다.
- 예시 키:
  - `rt:user:{userId}:sids`
- `issue`, `rotate`, `revoke` 시 인덱스를 함께 관리한다.
- 비밀번호 재설정 성공 시 `revokeAllByUserId(userId)`를 호출한다.

### 권장 2차 구현

- 필요하면 `token_version` 또는 `password_changed_at` 기반 접근 토큰 무효화도 도입한다.
- 1차에서는 Access Token 짧은 TTL에 의존해도 된다.

## 권장 엔드포인트

- `POST /api/auth/password-resets/request`
- `POST /api/auth/password-resets/confirm`

## 요청 예시

### 재설정 요청

```json
{
  "email": "user@example.com"
}
```

### 재설정 확정

```json
{
  "token": "base64url-token",
  "newPassword": "new-password-value",
  "confirmPassword": "new-password-value"
}
```

## 추가 보안 권장

- 이메일/IP 기준 rate limit
- 재설정 성공 알림 메일 발송
- 자동 로그인 금지
- 토큰 재사용 시도 기록
- 관리자/운영 로그 남김
