# 이메일 인증 설계

## 목표

- 로컬 회원가입 계정의 이메일 소유권을 검증한다.
- 토큰 탈취, 재사용, 대량 발송, 계정 존재 여부 노출을 줄인다.
- 향후 이메일 변경 인증에도 재사용 가능한 구조로 설계한다.

## 적용 범위

- 1차 범위: 회원가입 후 이메일 인증
- 2차 확장: 이메일 변경 인증

## 권장 모델

- `tb_users.is_email_verified`는 유지한다.
- 별도 토큰 테이블을 추가한다.
- `enabled`와 `locked`는 관리자/운영 상태로 계속 사용하고, 이메일 미인증 여부는 `is_email_verified`로만 판단한다.

## 토큰 저장 정책

- 메일에 담기는 토큰은 충분히 긴 랜덤 문자열을 사용한다.
- DB에는 토큰 원문을 저장하지 않고 `SHA-256` 해시만 저장한다.
- 토큰은 1회용이다.
- 새 토큰을 발급하면 이전 미사용 토큰은 즉시 만료 처리한다.

## 권장 토큰 속성

- 길이: 32바이트 이상 랜덤
- 만료: `24시간`
- 사용 횟수: 1회
- 저장값: `token_hash`
- 링크 예시: `https://app.mocktalk.site/verify-email?token=...`

## 메일 발송 정책

- 가입 성공 응답과 메일 발송은 분리한다.
- API 응답은 메일 발송 성공 여부와 분리하고, 발송 실패는 재시도 가능한 비동기 작업으로 처리한다.
- 메일 제목과 본문에는 불필요한 개인정보를 포함하지 않는다.

## 재발송 정책

- 재발송 API는 계정 존재 여부를 숨긴다.
- 권장 제한:
  - 동일 계정/이메일 기준 60초 쿨다운
  - 시간당 5회
  - 일별 10회
  - IP 기준 추가 제한
- 이미 인증된 계정이면 실제 발송 없이도 동일한 성공 응답을 반환한다.

## 인증 완료 정책

- 유효한 토큰이면 아래를 수행한다.
  - 토큰 `consumed_at` 기록
  - 사용자 `is_email_verified=true`
  - 다른 활성 인증 토큰 만료 처리
- 첫 구현은 자동 로그인 없이 종료한다.
- 성공 후 프론트는 로그인 페이지로 이동시키고 성공 메시지를 보여준다.

## 로그인 차단 정책

- 로컬 계정 로그인에서 비밀번호가 맞아도 `is_email_verified=false`이면 토큰을 발급하지 않는다.
- 응답 코드는 `403`을 권장한다.
- 응답 예시:
  - 코드: `AUTH_EMAIL_VERIFICATION_REQUIRED`
  - 메시지: `이메일 인증 후 로그인할 수 있습니다.`

## 권장 엔드포인트

- `POST /api/auth/email-verifications/resend`
- `POST /api/auth/email-verifications/confirm`

## 권장 요청 예시

### 재발송

```json
{
  "email": "user@example.com"
}
```

### 인증 확인

```json
{
  "token": "base64url-token"
}
```

## 보안 포인트

- 토큰 원문 미저장
- 1회용
- TTL 만료
- 최신 토큰만 유효
- 재발송 쿨다운
- IP 및 이메일 단위 rate limit
- 감사 로그 또는 운영 로그 남김

## 운영 로그 권장 항목

- 발급 시각
- 대상 사용자 ID
- 대상 이메일 스냅샷
- 발송 채널
- 성공/실패 결과
- 소비 시각
- 만료/무효/재사용 시도
