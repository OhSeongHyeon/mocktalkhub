# 회원가입 플로우

## 목표

- 민감정보를 추가로 받지 않으면서 운영 가능한 가입 흐름으로 전환한다.
- 로컬 계정은 이메일 검증이 끝나야 로그인 가능하도록 만든다.
- 약관 동의, 재발송, 예외 응답, 만료 계정 정리 기준을 문서화한다.

## 현재 상태

- 프론트 `/join` 페이지에서 `loginId`, `email`, `password`, `confirmPassword`, `userName`, `displayName`, `handle`을 전송한다.
- 백엔드 `/api/auth/join`은 중복 검사 후 `tb_users`에 즉시 저장한다.
- 저장 직후 `is_email_verified=false`지만 `is_enabled=true`이며, 로그인 시 이메일 인증 여부 검사가 없다.

## 권장 입력 항목

### 필수

- 로그인 아이디
- 이메일
- 비밀번호
- 비밀번호 확인
- 표시명 또는 서비스 내 사용자명
- 이용약관 동의
- 개인정보 수집/이용 동의
- 만 14세 이상 확인

### 선택

- 핸들 직접 입력
- 마케팅 정보 수신 동의

## `userName` 필드 정리 권장

- 현재 스키마와 주석은 `user_name`을 실명 의미로 설명하고 있다.
- 프로젝트 원칙이 민감정보 최소 수집이므로, 실명 수집 필드는 지양하는 편이 맞다.
- 우선순위는 아래와 같다.

1. 가장 권장: 가입 화면에서 실명 의미의 필드를 제거하고 `displayName`만 받는다.
2. 차선: `userName`의 의미를 "서비스 내부 사용자명"으로 재정의하고, UI 문구에서 실명 뉘앙스를 제거한다.
3. 비권장: 현재처럼 실명처럼 보이는 필드를 계속 필수 수집한다.

## 권장 상태 전이

- `가입 요청`
  - 유효성 검사
  - 중복 검사
  - 약관 동의 저장
  - 사용자 생성
  - 이메일 인증 토큰 생성 및 발송
- `이메일 미인증`
  - 로그인 불가
  - 재발송 가능
  - 일정 기간 후 미인증 계정 정리 가능
- `이메일 인증 완료`
  - `is_email_verified=true`
  - 일반 로그인 허용
- `잠금/비활성/탈퇴`
  - 기존 정책 유지

## 권장 UX 흐름

1. 사용자가 가입 폼과 필수 동의를 제출한다.
2. 서버는 계정을 생성하고 인증 메일 발송을 예약한다.
3. 응답은 "인증 메일을 확인해주세요"로 안내한다.
4. 사용자는 로그인 화면 또는 인증 안내 화면으로 이동한다.
5. 사용자가 메일 링크를 클릭하면 프론트 인증 완료 페이지를 거쳐 백엔드 확인 API를 호출한다.
6. 성공 시 로그인 페이지로 이동시키고, 첫 구현에서는 자동 로그인하지 않는다.

## 로그인 정책

- 로컬 계정에서 `is_email_verified=false`이면 로그인 성공 대신 `403 EMAIL_VERIFICATION_REQUIRED`를 반환한다.
- 응답 본문에는 재발송을 유도할 수 있는 안내 메시지를 포함한다.
- 소셜 로그인 계정은 제공자 이메일 신뢰 정책에 따라 기존 동작을 유지하되, 이메일이 없는 제공자에 대한 예외 정책은 별도 검토한다.

## 예외 처리 원칙

- 회원가입 자체에서는 입력 검증 오류를 구체적으로 알려도 된다.
- 다만 이메일 인증 재발송, 비밀번호 재설정 요청처럼 계정 존재 여부가 드러나면 안 되는 API는 항상 동일한 성공 메시지를 반환한다.
- 미인증 계정 로그인 차단은 보안과 UX를 위해 명시적 에러 코드로 응답한다.

## 미인증 계정 정리 기준

- 가입 후 `7일` 또는 `14일` 내 미인증이면 정리 대상으로 본다.
- 정리 방식은 하드 삭제보다 "완전 삭제 가능 시 삭제, 아니면 비활성 후 별도 배치 정리"로 단순화한다.
- 초기 구현은 배치 문서화만 하고, 실제 삭제는 2차 작업으로 미뤄도 된다.
