# 콘텐츠 / 환율·금 시세 요구사항 정의서

## 문서 목적

- 사이드 메뉴 `이미지 갤러리`를 `콘텐츠`로 개편하는 요구사항을 정리한다.
- `콘텐츠` 허브와 `환율/금 시세` 화면의 사용자 범위, 기술 제약, 운영 정책을 명확히 한다.
- 데이터 축적, 수동 최신화, 과거 데이터 임포트까지 포함한 운영 방향의 기준 문서로 사용한다.

## 1. 배경

현재 사이드 메뉴에는 `이미지 갤러리` 항목이 있으나 별도 화면과 기능이 구현되어 있지 않다.

이번 작업에서는 이를 `콘텐츠` 진입점으로 확장하고, 그 안에서 외부 시세 기반 정보성 콘텐츠를 제공한다.

## 2. 요구사항 요약

### 2.1 메뉴 개편

- 사이드 메뉴의 `이미지 갤러리` 문구를 `콘텐츠`로 변경한다.
- `콘텐츠` 클릭 시 신규 허브 화면으로 이동한다.

### 2.2 콘텐츠 허브 구성

허브 화면에는 아래 두 콘텐츠 카드를 제공한다.

- `이미지 갤러리`
  - 준비중 상태로 노출
- `환율/금 시세`
  - 실제 진입 가능한 콘텐츠로 제공

### 2.3 환율/금 시세 콘텐츠

- 환율과 금 시세를 하나의 화면에서 함께 제공한다.
- 사용자는 전체 흐름과 개별 종목 흐름을 모두 확인할 수 있어야 한다.
- 거래 기능은 필요하지 않다.
- 값의 변동 추이와 현재 상태를 확인할 수 있으면 충분하다.

### 2.4 운영 제어 기능

- 운영자는 백오피스에서 시세 수집을 수동으로 다시 실행할 수 있어야 한다.
- 운영자는 백오피스에서 과거 시세 데이터를 파일로 임포트할 수 있어야 한다.
- 과거 데이터 채우기는 외부 API 기간 재조회보다 `CSV/XLSX 임포트`를 우선한다.

## 3. 핵심 요구사항

### 3.1 데이터 호출 구조

- 프론트에서 외부 OpenAPI를 직접 호출하지 않는다.
- 반드시 `프론트 -> 백엔드 -> 외부 API` 구조를 따른다.
- 외부 API 응답은 백엔드가 가공하고 필요한 데이터만 프론트에 반환한다.

### 3.2 비용 제약

- 무과금이 원칙이다.
- 가능하면 회원가입이나 API 키 발급 없이 사용할 수 있는 데이터 소스를 우선 검토한다.
- 회원가입 또는 API 키가 필요하더라도 무료 범위 안에서 유지 가능하면 허용한다.
- 유료 플랜 전제의 구조는 채택하지 않는다.

### 3.3 사용자 범위

- 비로그인 사용자도 접근 가능한 공개 콘텐츠로 제공한다.
- 인증 의존 기능은 이번 범위에 포함하지 않는다.

### 3.4 그래프 요구

- 긴 갱신주기의 시세 그래프를 제공한다.
- 사용자는 최소 `7일`, `30일`, `1년` 흐름을 볼 수 있어야 한다.
- 사용자는 `시작일`, `종료일`을 직접 선택해 범위를 지정할 수 있어야 한다.
- 그래프는 참고용 정보 제공 목적이며 투자 의사결정 기능을 제공하지 않는다.

### 3.5 초기 데이터 정책

- 1차 범위에서는 외부 API 기반 백필을 포함하지 않는다.
- 시세 데이터는 배포 시점부터 자동 축적한다.
- 과거 데이터가 필요하면 백오피스에서 `CSV/XLSX 임포트`로 채운다.

## 4. 세부 요구사항

### 4.1 메뉴 / 라우팅

- 사이드 메뉴 항목명 변경
- `콘텐츠` 허브 신규 라우트 추가
- `환율/금 시세` 상세 화면 라우트 추가

### 4.2 허브 화면

- 카드형 또는 섹션형 구성
- `이미지 갤러리`는 준비중 뱃지와 설명 문구 노출
- `환율/금 시세`는 진입 버튼 또는 링크 제공

### 4.3 환율/금 시세 화면

- 전체 종목 통합 그래프
- 종목 선택 탭
- 현재 값 요약
- 직접 선택 가능한 기간 제어
- 마지막 갱신 시각
- 데이터 기준 통화/단위 표시

### 4.4 백오피스 운영 기능

- `지금 최신화`
  - 현재 시세를 즉시 1회 수집
- `CSV/XLSX 임포트`
  - 운영자가 과거 데이터를 파일로 업로드
  - 종목별 또는 통합 파일 형식을 허용할 수 있다

## 5. 비범위

이번 범위에 포함하지 않는 항목은 아래와 같다.

- 실제 환전, 매수/매도, 주문 기능
- 자산 계산기
- 사용자 알림 설정
- 구독, 관심종목 저장
- 푸시형 실시간 체결 정보
- 실제 이미지 갤러리 기능 구현
- 외부 API 기간 재조회 방식의 백필

## 6. 기술/운영 제약

### 6.1 데이터 정확도

- 데이터는 참고용 정보다.
- 원천 데이터 공급자의 갱신 주기와 지연을 그대로 가질 수 있다.
- 금 시세는 실제 현물 대신 참고용 국제 금 시세 또는 금 선물 시세일 수 있다.

### 6.2 외부 API 안정성

- 외부 API 장애가 서비스 전체 장애로 전파되면 안 된다.
- 시세 수집 실패 시 마지막 정상 스냅샷을 기반으로 화면을 유지할 수 있어야 한다.

### 6.3 인프라 비용

- 현재 인프라 구조에서 별도 상용 메시지 브로커, 유료 시세 피드, 별도 분석 인프라 도입은 하지 않는다.

## 7. 데이터 갱신 전략 요구

실시간 스트리밍보다 운영 단순성과 무과금 유지가 우선이다.

요구 수준:

- 사용자는 최신에 가까운 값을 볼 수 있어야 한다.
- 초단위 tick 데이터는 필요하지 않다.
- 백엔드 수집은 환율/금 시세 모두 하루 1회를 기본으로 한다.
- 프론트는 화면 진입 중일 때만 저빈도 갱신을 수행할 수 있다.
- 과거 데이터 보강은 스케줄러가 아니라 파일 임포트로 처리한다.

## 8. 인터랙션 요구

- `콘텐츠` 메뉴 진입
- 허브에서 `환율/금 시세` 선택
- 종목 탭 전환
- 기간 프리셋 전환
- 시작일/종료일 직접 선택
- 최신 데이터 재조회 또는 자동 갱신

## 9. 성공 기준

- 사용자가 `콘텐츠` 메뉴에서 준비중 콘텐츠와 실제 제공 콘텐츠를 구분할 수 있다.
- `환율/금 시세` 화면에서 현재값과 그래프를 무리 없이 확인할 수 있다.
- 외부 API를 프론트가 직접 호출하지 않는다.
- 무과금 또는 무료 범위로 운영 가능하다.
- 시세 기능 장애가 게시판/인증/기존 홈 기능에 영향을 주지 않는다.
- 운영자가 백오피스에서 수동 최신화와 CSV/XLSX 임포트를 수행할 수 있다.

## 10. 권장 구현 원칙

- 첫 단계는 `가벼운 정보성 콘텐츠`로 시작한다.
- 무료 데이터 소스의 한계를 제품 문구와 갱신 정책으로 흡수한다.
- 실시간성보다 안정성, 단순성, 유지보수성을 우선한다.
- 자동 수집과 과거 데이터 임포트의 책임을 분리한다.
