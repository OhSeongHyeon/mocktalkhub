# News Bot Requirements

## 1. 문서 목적

이 문서는 외부 공개 API를 주기적으로 호출해 새소식을 수집하고, 이를 Mocktalk 게시판/게시글로 적재하는 `news-bot` 기능의 요구사항을 정리한다.

정리 기준일은 `2026-03-15`이며, 현재 `mocktalkback` 코드베이스를 확인한 결과를 함께 반영한다.

외부 소스별 운영 가이드는 `external-sources-guide.md`를 참고한다.
처음 등록용 예시 템플릿은 `starter-templates.md`를 참고한다.

## 2. 배경

- 초반 커뮤니티 활성화를 위해 게시판과 게시글을 자동으로 채울 수단이 필요하다.
- 수집 작업은 `n8n`이 아니라 백엔드가 직접 외부 API를 호출하는 방식이어야 한다.
- 운영자는 백오피스에서 기능 on/off, 수집 주기, 즉시 실행을 제어할 수 있어야 한다.
- 외부 API는 회원가입이나 API 키 발급 없이 사용할 수 있어야 한다.

## 3. 현재 코드베이스 확인 결과

- `tb_articles.user_id`는 필수다. 외부 수집 글도 반드시 내부 사용자 한 명의 소유로 저장되어야 한다.
- 뉴스봇은 `@Scheduled(fixedDelayString = "${app.news-bot.dispatcher-interval-ms:60000}")` 기반 dispatcher poller와 `next_run_at` 기준 due 잡 조회 구조로 구현되어 있다.
- 뉴스봇 전용 테이블 `tb_news_collection_jobs`, `tb_news_collected_items`가 추가되어 잡 설정과 dedupe 상태를 DB에 저장한다.
- `ArticleImportService`에는 `ArticleCategoryRepository.findByBoardIdAndCategoryNameIgnoreCase(...)`와 `ensureCategory(...)`가 있어 카테고리 자동 생성 패턴은 일부 재사용 가능하다.
- 외부 API 호출은 `RestClient` 기반으로 구현되어 있다.
- 관리자 API와 최소 백오피스 화면이 구현되어 있으며, 목록/생성/수정/on-off/즉시 실행이 가능하다.
- 백오피스 폼은 소스별 preset, 조건부 입력, 필수값 하이라이트를 지원한다.

## 4. 범위

### 포함

- 외부 새소식 수집용 잡(Job) 등록/수정/비활성화
- 주기적 수집 및 즉시 실행
- 게시판/카테고리 자동 생성
- 게시글 자동 생성 및 동일 소스 항목 중복 방지
- 백오피스용 관리자 API
- 최소 백오피스 화면
- 운영/장애 추적을 위한 실행 상태 저장

### 제외

- 외부 API 키 발급이 필요한 공급자 연동
- 댓글, 반응, 알림 자동 생성
- 전문 번역/요약 AI 파이프라인
- n8n, Airflow 같은 외부 자동화 도구 연동
- 실행 단위 상세 이력 전용 화면

## 5. 기능 요구사항

### FR-01. 수집 잡 등록

- 운영자는 새소식 수집 잡을 등록할 수 있어야 한다.
- 잡에는 최소한 다음 정보가 필요하다.
- 소스 유형
- 소스 식별값 또는 URL
- 대상 게시판 slug/name
- 기본 카테고리명
- 수집 주기
- on/off 상태
- 회당 수집 개수 제한
- 게시판/카테고리 자동 생성 여부

### FR-02. 백오피스 제어

- 운영자는 잡을 활성화/비활성화할 수 있어야 한다.
- 운영자는 잡 주기를 변경할 수 있어야 한다.
- 운영자는 마지막 실행 시각, 마지막 성공 시각, 마지막 실패 메시지를 확인할 수 있어야 한다.
- 운영자는 잡을 즉시 1회 실행할 수 있어야 한다.

### FR-03. 주기 실행

- 백엔드가 주기적으로 due 상태의 잡을 찾아 실행해야 한다.
- 예시 주기:
- 1시간마다
- 24시간마다
- 운영자가 설정한 간격 기준

### FR-04. 외부 API 제약

- 회원가입이나 API 키 발급이 없어야 한다.
- `User-Agent` 정도의 기본 헤더는 허용한다.
- 공용 공개 데이터만 수집 대상으로 삼는다.

### FR-05. 자동 생성

- 대상 게시판이 없으면 자동 생성할 수 있어야 한다.
- 대상 카테고리가 없으면 자동 생성할 수 있어야 한다.
- 자동 생성 여부는 잡 단위 설정으로 제어할 수 있어야 한다.

### FR-06. 게시글 생성

- 수집한 항목은 게시글로 저장되어야 한다.
- 게시글 작성자는 시스템 전용 계정 `news_bot`이어야 한다.
- 게시글 본문에는 최소한 제목, 원문 링크, 발행 시각, 출처명이 포함되어야 한다.
- 게시글 포맷은 기본적으로 `MARKDOWN`을 사용한다.

### FR-07. 중복 방지

- 같은 외부 항목은 반복 실행 시 중복 생성되면 안 된다.
- 동일 항목이 수정된 경우에는 `skip` 또는 `update` 정책 중 하나로 처리해야 한다.
- MVP는 `동일 key + 동일 내용 = skip`, `동일 key + 내용 변경 = update`를 기본 정책으로 둔다.

### FR-08. 소스별 지원 범위

- 다음 소스 유형을 지원 대상으로 둔다.
- Hacker News
- DEV API
- GitHub Releases API
- RSS/Atom feed

### FR-09. 관리자 권한

- 잡 생성/수정/on-off/즉시 실행은 `ADMIN` 또는 `MANAGER` 권한으로 제한한다.
- 현재 구현도 `MANAGER`, `ADMIN` 모두 접근 가능한 구조다.
- 운영 제어자는 `admin` 계정이고, 실제 뉴스 게시글 작성자는 `news_bot` 계정으로 분리한다.

## 6. 비기능 요구사항

### NFR-01. 안정성

- 외부 API 실패가 전체 애플리케이션 기동을 막으면 안 된다.
- 한 잡 실패가 다른 잡 실행을 막으면 안 된다.

### NFR-02. 성능

- 한 번의 스케줄 실행은 짧게 끝나야 한다.
- 외부 호출을 포함한 긴 작업을 하나의 DB 트랜잭션으로 묶지 않는다.

### NFR-03. 운영성

- 잡 상태와 마지막 오류가 DB에 남아야 한다.
- 관리자 액션은 기존 `tb_admin_audit_logs`에 남길 수 있어야 한다.
- 뉴스봇 기능은 `.env.{profile}`에 별도 값을 추가하지 않아도 동작해야 한다.
- 환경변수는 필수 설정이 아니라 전역 안전장치나 공통 네트워크 설정 override 용도로만 사용한다.

### NFR-04. 확장성

- 새 소스 유형을 추가할 때 기존 잡 실행 코어 로직은 최대한 건드리지 않아야 한다.
- 소스별 어댑터를 교체 가능하게 분리해야 한다.

### NFR-05. 테스트 가능성

- 외부 API를 직접 호출하는 통합 테스트 대신, fixture 기반 어댑터 테스트를 우선한다.
- 스케줄러는 실제 시간 의존 대신 `Clock` 주입 또는 시각 계산 분리로 테스트 가능해야 한다.

## 7. 새 테이블 필요성 판단

요구사항상 `새 테이블 없이 끝내는 안`은 권장하지 않는다.

이유는 아래와 같다.

- 백오피스에서 바꾸는 on/off, 주기, 마지막 실행 시각을 `application.yml`만으로는 저장할 수 없다.
- 중복 방지를 하려면 외부 항목 key와 내부 article_id 연결 정보를 별도로 보관해야 한다.
- 게시글 본문 안에 원문 URL만 넣고 중복을 판별하는 방식은 조회 비용과 정확도 측면에서 취약하다.

따라서 최소 2개 테이블이 필요하다.

- 잡 정의/상태 저장 테이블
- 수집 항목 dedupe 및 동기화 상태 저장 테이블

## 8. 외부 API 후보 검증 메모

`2026-03-15` 기준으로 아래 후보의 무인증 호출 가능성을 확인했다.

| 후보 | 확인 결과 | 비고 |
| --- | --- | --- |
| Hacker News API | 공개 호출 가능 | `topstories.json` 응답 `200` 확인 |
| DEV API | 공개 호출 가능 | `User-Agent` 포함 요청 응답 `200` 확인 |
| GitHub Releases API | 공개 호출 가능 | 응답 `200`, 무인증 rate limit `60/hour` 확인 |
| 공식 RSS/Atom | 공개 호출 가능 | 예시로 `https://spring.io/blog.atom` 응답 `200` 확인 |

운영 우선순위는 아래처럼 둔다.

- 1순위: GitHub Releases, 공식 RSS/Atom
- 2순위: Hacker News
- 3순위: DEV API

판단 이유는 아래와 같다.

- GitHub Releases와 RSS는 제품 업데이트/공식 블로그성 콘텐츠라 게시판 적재 품질이 높다.
- Hacker News는 링크 모음형이라 중복 기사와 외부 링크 편중 가능성이 있다.
- DEV API는 공개 호출은 가능하지만 운영 정책 변경 가능성을 더 주의해야 한다.

## 9. 핵심 제약 및 의사결정

- 게시글은 반드시 내부 사용자 소유여야 하므로 시스템 계정 `news_bot`이 필요하다.
- `admin`은 백오피스에서 뉴스봇 잡을 생성/수정/실행하는 제어자이고, `news_bot`은 실제 게시글 작성자다.
- `news_bot`은 일반 회원가입으로 만드는 계정이 아니라 seed 또는 초기화 단계에서 보장 생성하는 시스템 계정으로 본다.
- 자동 생성되는 새소식 게시판은 기본적으로 `PUBLIC + OWNER 쓰기 정책`을 권장한다.
- 최소 백오피스 화면과 관리자 API를 함께 운영 기준으로 둔다.
- 스케줄 제어는 정적 cron 프로퍼티가 아니라 DB 기반 잡 설정으로 처리해야 한다.
- 환경변수는 선택적으로 지원하되, 잡별 on/off와 주기 제어는 백오피스/DB를 기준으로 한다.
