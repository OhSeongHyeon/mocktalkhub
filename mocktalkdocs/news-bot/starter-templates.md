# News Bot Starter Templates

## 1. 문서 목적

이 문서는 뉴스봇을 처음 사용하는 운영자가 백오피스에서 바로 따라 입력할 수 있도록 만드는 스타터 가이드다.

상세한 소스별 설명과 주의점은 `external-sources-guide.md`를 참고한다. 이 문서는 설명서가 아니라 `첫 잡을 빠르게 만드는 체크리스트`에 가깝다.

## 2. 먼저 권장하는 시작 방식

처음부터 많은 잡을 만들지 않는 편이 안전하다. 아래 순서로 시작하는 것을 권장한다.

1. `주 잡 1개`만 먼저 만든다.
2. 실제로 게시글이 잘 들어오는지 확인한다.
3. 게시판 분위기와 글 품질이 맞는지 본다.
4. 괜찮으면 `보조 잡 1개`를 추가한다.

초기 권장 원칙:

- `1 job = 1 board` 기준으로 운영
- `게시판 자동 생성`은 기본적으로 끄기
- `카테고리 자동 생성`은 켜기
- 주기는 짧게 잡지 않기
- 한 번에 가져오는 글 수는 작게 시작하기

## 3. 백오피스 입력 전에 정할 것

백오피스에서 입력하기 전에 아래 3가지만 먼저 정하면 훨씬 쉽다.

### 3.1 어떤 종류의 소식을 모을지

- 공식 릴리즈 공지
- 공식 블로그 글
- 개발 아티클
- 기술 커뮤니티 글

### 3.2 어느 게시판에 넣을지

- 예: `spring-news`
- 예: `backend-news`
- 예: `infra-news`

### 3.3 어떤 카테고리로 묶을지

- 예: `release`
- 예: `article`
- 예: `official`
- 예: `community`

## 4. 백오피스 입력 순서

실제 화면에서는 아래 순서대로 보면 된다.

### 4.1 공통 정보

- `잡 이름`
- `외부 소스`

### 4.2 외부 소스 조건

소스에 따라 아래 중 일부만 입력한다.

- `Hacker News`
  - `storyType`
- `DEV API`
  - `수집 기준`
  - `tag` 또는 `username`
- `GitHub Releases`
  - `owner`
  - `repo`
- `RSS / Atom`
  - `feedUrl`

### 4.3 내부 적재 정보

- `대상 게시판 slug`
- `자동 생성용 게시판 이름`
- `기본 카테고리`

### 4.4 실행 정책

- `수집 주기(분)`
- `1회 최대 수집 건수`
- `timezone`

### 4.5 자동 생성 정책

- `게시판 자동 생성 허용`
- `카테고리 자동 생성 허용`

## 5. 스타터 1. 스프링 새소식

가장 무난한 첫 스타터다. 공식 업데이트 성격이 강해서 품질이 안정적이다.

추천 소스:

- `GitHub Releases`

권장 입력값:

- `잡 이름`: `스프링부트 릴리즈`
- `외부 소스`: `GitHub Releases`
- `owner`: `spring-projects`
- `repo`: `spring-boot`
- `대상 게시판 slug`: `spring-news`
- `자동 생성용 게시판 이름`: `스프링 새소식`
- `기본 카테고리`: `release`
- `수집 주기(분)`: `720`
- `1회 최대 수집 건수`: `1`
- `게시판 자동 생성 허용`: `끔`
- `카테고리 자동 생성 허용`: `켬`

이 스타터가 좋은 이유:

- 공식 릴리즈만 가져와서 품질이 안정적이다.
- 글 수가 많지 않아 초반 운영 부담이 적다.
- 중복과 노이즈가 적다.

이런 경우에 적합하다:

- 공식 제품 업데이트 게시판이 필요할 때
- 운영 초반에 품질 위주로 채우고 싶을 때

## 6. 스타터 2. 백엔드 새소식

개발 아티클 기반으로 게시판을 채우는 기본형이다.

추천 소스:

- `DEV API`

권장 입력값:

- `잡 이름`: `백엔드 새소식`
- `외부 소스`: `DEV API`
- `수집 기준`: `태그 기준`
- `tag`: `backend`
- `대상 게시판 slug`: `backend-news`
- `자동 생성용 게시판 이름`: `백엔드 새소식`
- `기본 카테고리`: `article`
- `수집 주기(분)`: `180`
- `1회 최대 수집 건수`: `10`
- `게시판 자동 생성 허용`: `끔`
- `카테고리 자동 생성 허용`: `켬`

이 스타터가 좋은 이유:

- 주제가 비교적 선명하다.
- 너무 좁지도 넓지도 않은 편이라 초반 볼륨 확보에 좋다.
- 게시판이 완전히 비어 보이는 상황을 줄이는 데 도움이 된다.

주의:

- `spring-news` 같은 게시판 slug를 `tag`에 넣는 것이 아니다.
- `tag`는 외부 검색 조건이고, `대상 게시판 slug`는 내부 저장 위치다.

## 7. 스타터 3. 인프라 새소식

공식 블로그 기반으로 운영하고 싶을 때 적합하다.

추천 소스:

- `RSS / Atom`

권장 입력값:

- `잡 이름`: `스프링 공식 블로그`
- `외부 소스`: `RSS / Atom`
- `feedUrl`: `https://spring.io/blog.atom`
- `대상 게시판 slug`: `infra-news`
- `자동 생성용 게시판 이름`: `인프라 새소식`
- `기본 카테고리`: `official`
- `수집 주기(분)`: `360`
- `1회 최대 수집 건수`: `8`
- `게시판 자동 생성 허용`: `끔`
- `카테고리 자동 생성 허용`: `켬`

이 스타터가 좋은 이유:

- 공식 채널 기반이라 품질이 비교적 높다.
- DEV보다 의견성 글이 적고, 공식 발표나 기술 블로그 성격이 강하다.

주의:

- `infra-news` 게시판을 쓰더라도 실제 피드가 스프링 블로그면 내용은 스프링 중심이다.
- 게시판 이름과 피드 주제가 너무 멀어지지 않게 맞추는 편이 좋다.

## 8. 스타터 4. 프론트엔드 새소식

프론트엔드 전반 소식을 공식 채널 중심으로 가져오고 싶을 때 적합하다.

추천 소스:

- `RSS / Atom`

권장 입력값:

- `잡 이름`: `프론트엔드 새소식`
- `외부 소스`: `RSS / Atom`
- `feedUrl`: `https://web.dev/static/blog/feed.xml`
- `대상 게시판 slug`: `frontend-news`
- `자동 생성용 게시판 이름`: `프론트엔드 새소식`
- `기본 카테고리`: `official`
- `수집 주기(분)`: `360`
- `1회 최대 수집 건수`: `8`
- `게시판 자동 생성 허용`: `끔`
- `카테고리 자동 생성 허용`: `켬`

이 스타터가 좋은 이유:

- `web.dev`는 웹 플랫폼, 성능, 접근성, 브라우저 기능 업데이트를 다루는 공식 채널이라 품질이 안정적이다.
- React, Vue, CSS, 성능, Baseline 같은 프론트엔드 공통 주제를 두루 다룬다.
- 운영 초반에 게시판 성격을 흐리지 않고 시작하기 좋다.

주의:

- 이 스타터는 `프레임워크 단일 소식`보다 `웹 플랫폼 전반`에 가깝다.
- Vue나 React만 따로 모으고 싶다면 이후에 별도 잡으로 분리하는 편이 낫다.

## 9. 스타터 5. PostgreSQL 새소식

PostgreSQL 관련 글을 공식 커뮤니티 생태계 기준으로 넓게 모으고 싶을 때 적합하다.

추천 소스:

- `RSS / Atom`

권장 입력값:

- `잡 이름`: `PostgreSQL 새소식`
- `외부 소스`: `RSS / Atom`
- `feedUrl`: `https://planet.postgresql.org/rss20.xml`
- `대상 게시판 slug`: `postgres-news`
- `자동 생성용 게시판 이름`: `PostgreSQL 새소식`
- `기본 카테고리`: `community`
- `수집 주기(분)`: `360`
- `1회 최대 수집 건수`: `8`
- `게시판 자동 생성 허용`: `끔`
- `카테고리 자동 생성 허용`: `켬`

이 스타터가 좋은 이유:

- Planet PostgreSQL은 PostgreSQL 커뮤니티가 운영하는 블로그 집계 서비스라 관련 글 볼륨을 확보하기 좋다.
- 운영, 튜닝, 확장, 릴리즈 해설, 생태계 도구 글이 함께 들어와서 게시판을 채우기 좋다.
- 공식 릴리즈 공지보다 글 수가 안정적으로 나온다.

주의:

- 이 스타터는 `공식 릴리즈 공지 전용`이 아니라 `PostgreSQL 생태계 소식`에 가깝다.
- 글 품질은 전반적으로 괜찮지만, 운영 팁/커뮤니티 글/도구 소개가 함께 섞일 수 있다.

권장 입력값:

- `잡 이름`: `테크 커뮤니티 새소식`
- `외부 소스`: `Hacker News`
- `storyType`: `topstories`
- `대상 게시판 slug`: `tech-news`
- `자동 생성용 게시판 이름`: `테크 새소식`
- `기본 카테고리`: `community`
- `수집 주기(분)`: `180`
- `1회 최대 수집 건수`: `10`
- `게시판 자동 생성 허용`: `끔`
- `카테고리 자동 생성 허용`: `켬`

주의:

- Hacker News는 기술 글 외에 스타트업, 서비스 운영, 창업 관련 글도 섞일 수 있다.
- 운영 초반 메인 게시판 하나를 이 소스로만 채우면 게시판 성격이 흐려질 수 있다.

## 11. 처음엔 이 조합만 권장

처음에는 아래 둘 중 하나만 선택해도 충분하다.

### 조합 A. 안정형

- `스프링부트 릴리즈`
- `백엔드 새소식`

특징:

- 공식 업데이트와 일반 개발 아티클을 균형 있게 가져온다.
- 노이즈가 비교적 적다.

### 조합 B. 공식 채널형

- `스프링부트 릴리즈`
- `스프링 공식 블로그`

특징:

- 전체적으로 품질이 안정적이다.
- 게시글 수는 많지 않을 수 있다.

### 조합 C. 웹 개발형

- `프론트엔드 새소식`
- `백엔드 새소식`

특징:

- 프론트엔드와 백엔드 소식을 같이 채워서 개발 카테고리 균형을 맞추기 좋다.
- 둘 다 공식 문서형 채널만 쓰는 구조는 아니라 글 수 확보에 유리하다.

### 조합 D. 데이터베이스형

- `PostgreSQL 새소식`
- `스프링부트 릴리즈`

특징:

- 애플리케이션과 데이터베이스 소식을 같이 운영할 때 무난하다.
- PostgreSQL 게시판은 생태계 글 중심이라 운영/성능/도구 이야기가 함께 들어올 수 있다.

## 12. 스타터를 그대로 쓸 때 바꿔도 되는 값

아래 값은 운영 환경에 맞게 바꿔도 괜찮다.

- `잡 이름`
- `대상 게시판 slug`
- `자동 생성용 게시판 이름`
- `기본 카테고리`
- `수집 주기(분)`

아래 값은 처음에는 그대로 두는 편이 안전하다.

- `GitHub Releases.owner = spring-projects`
- `GitHub Releases.repo = spring-boot`
- `DEV.tag = backend`
- `RSS.feedUrl = https://spring.io/blog.atom`
- `RSS.feedUrl = https://web.dev/static/blog/feed.xml`
- `RSS.feedUrl = https://planet.postgresql.org/rss20.xml`
- `Hacker News.storyType = topstories`

## 13. 첫 등록 후 확인 체크리스트

잡을 만든 뒤에는 아래를 확인한다.

1. 잡이 저장되었는가
2. 즉시 실행 또는 다음 실행에서 실제 글이 들어오는가
3. 게시판 위치가 의도한 곳이 맞는가
4. 카테고리명이 운영 기준과 맞는가
5. 글 수가 너무 많거나 너무 적지 않은가
6. 게시글 제목과 링크 품질이 게시판 성격에 맞는가

## 14. 실행 중 수정/비활성화에 대한 운영 기준

운영자가 잡을 제어하는 권한은 유지한다. 따라서 `running` 중에도 수정이나 비활성화는 할 수 있다.

다만 아래처럼 이해하는 것이 맞다.

- 이미 시작된 실행 1회는 즉시 중단되지 않는다.
- 비활성화해도 현재 돌고 있는 실행은 보통 끝까지 간다.
- 수정한 주기, 대상 게시판, 카테고리, fetch limit 같은 값은 다음 실행부터 반영된다.
- `지금 실행`과 자동 스케줄이 같은 타이밍에 겹치면 중복 실행은 막힌다.

## 15. 막히기 쉬운 포인트

### 15.1 DEV에서 글이 안 들어오는 경우

- `태그 기준`인데 `tag`가 비어 있음
- `작성자 기준`인데 `username`이 틀림
- `대상 게시판 slug`와 `tag`를 헷갈림

### 15.2 GitHub Releases에서 결과가 너무 적은 경우

- 원래 최신 릴리즈만 보는 구조라 한 번에 많이 나오지 않을 수 있다
- `1회 최대 수집 건수`를 늘려도 소스 특성상 큰 차이가 없을 수 있다

### 15.3 RSS에서 글이 안 들어오는 경우

- 일반 웹페이지 주소를 넣고 실제 RSS/Atom 피드 주소를 안 넣었을 수 있다
- 피드 주소는 보통 `.xml`, `.rss`, `.atom` 형태이지만 항상 그런 것은 아니다

### 15.4 게시판/카테고리가 이상하게 생기는 경우

- 자동 생성 옵션을 켠 상태로 운영 기준 없는 이름을 넣었을 수 있다
- 운영 초반에는 `게시판 자동 생성`을 꺼두는 편이 안전하다

### 15.5 프론트엔드 뉴스가 너무 넓게 느껴지는 경우

- `web.dev`는 웹 플랫폼 전반을 다루므로 특정 프레임워크 게시판과는 성격이 다를 수 있다
- 프론트엔드 게시판을 더 좁게 운영하고 싶다면 추후 Vue, React, CSS 같은 세부 잡으로 분리한다

### 15.6 PostgreSQL 뉴스가 공식 공지처럼만 보이지 않는 경우

- Planet PostgreSQL은 공식 릴리즈 전용 피드가 아니라 커뮤니티 집계 피드다
- 릴리즈 공지 전용 게시판이 필요하면 별도 공식 소스 잡을 추가하는 편이 낫다
