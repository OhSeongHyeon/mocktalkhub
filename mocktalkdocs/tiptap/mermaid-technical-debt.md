# Mermaid 최적화 기술부채 메모

## 문서 목적
- 목적: 현재 `Mermaid` 지원 상태와 추가 최적화를 당장 진행하지 않는 이유를 기술부채 관점에서 기록한다.
- 기준일: 2026-03-15
- 범위: `Markdown/WYSIWYG` 본문 내 `Mermaid` 렌더링과 프론트 번들 최적화 이슈만 다룬다.

## 관련 문서
- [Markdown 작성 UX 재설계안](./markdown-mode-design.md)
- [마크다운 대량 임포트 고도화 설계안](./markdown-bulk-import-advanced-design.md)

## 현재 상태 요약
- 본문 `Mermaid` 렌더링은 지원한다.
- 현재 지원 다이어그램은 `graph/flowchart`, `sequenceDiagram`, `erDiagram`이다.
- 미지원 다이어그램은 렌더 에러로 죽이지 않고 원본 코드블록 fallback으로 표시한다.
- `Mermaid`는 필요한 시점에만 동적 import 하도록 되어 있다.
- `vite.config.ts`의 수동 청크는 `vue`, `editor`, `sanitize` 계열만 명시하고, Mermaid 패키지는 강제 청크 분리하지 않는다.
- 번들 크기 수치는 빌드 결과에 따라 바뀌므로 이 문서에는 고정 수치를 남기지 않는다.

## 이번에 해결한 문제
- 기존 수동 청크 분리 과정에서 `vendor`와 `Mermaid parser` 계열이 서로 import 하는 순환 청크가 생겼다.
- 그 결과 배포 후 브라우저에서 `Cannot access 'Gn' before initialization` 런타임 오류가 발생했다.
- 현재는 Mermaid 관련 패키지를 억지로 별도 청크로 묶지 않고, Vite/Rollup 기본 분할에 맡겨 해당 장애를 해소했다.

## 왜 지금 추가 최적화를 멈추는가
- 현재 최우선 문제였던 `배포 후 화면 미노출`은 해결됐다.
- Mermaid는 동적 import로 필요한 시점에만 로드되고, 강제 청크 분리로 인한 순환 참조 문제도 제거됐다.
- 남은 최적화는 구현 복잡도 대비 즉시 사용자 가치가 크지 않다.
- 추가 최적화는 단순 번들 설정 수준이 아니라 렌더 구조 자체를 다시 고민해야 하는 영역이다.

## 현재 남겨둔 기술부채
- `Mermaid`가 실제 사용하는 다이어그램만 더 좁게 번들링하지 못했다.
- 상세 페이지 렌더는 아직 `사전 SVG 렌더링` 구조가 아니다.
- 지원하지 않는 다이어그램 코드 일부가 번들 결과에 여전히 포함될 수 있다.
- `Mermaid` 렌더 성능에 대한 실사용 계측 지표가 없다.
- 모바일 저성능 환경 기준 체감 성능 검증이 부족하다.

## 재개 조건
- `Mermaid`가 포함된 글 상세 페이지 첫 진입이 느리다는 실제 사용자 피드백이 쌓일 때
- 관리자/작성자 화면에서 `Mermaid` 사용 빈도가 높아질 때
- 모바일 환경에서 체감 지연이 재현될 때
- 트래픽 증가나 문서 수 증가로 `Mermaid` 렌더 비용이 문제로 드러날 때

## 재개 시 우선순위
1. 실제 사용 중인 다이어그램 타입만 남기고 지원 범위를 더 좁힐 수 있는지 재점검
2. 게시글 상세 페이지는 `Mermaid 런타임 렌더` 대신 `SVG 사전 렌더`로 전환 검토
3. `Mermaid` 대체 렌더러 또는 서버/사이드카 렌더러 도입 검토
4. 실사용 계측 추가
   - `Mermaid` 포함 글 상세 진입 시간
   - 작성 화면 첫 진입 시간
   - 모바일 환경 실패율

## 현재 운영 원칙
- 지금은 `기능 유지 > 추가 최적화` 우선순위를 따른다.
- `Mermaid`는 기술부채로 기록만 남기고, 기능 개발과 안정화 작업을 우선한다.
- 성능 이슈가 실제로 확인되면 그때 `SVG 사전 렌더`를 1순위 패치 후보로 본다.
