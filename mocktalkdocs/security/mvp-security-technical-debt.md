# MVP 보안 기술부채 정리

## 문서 목적
- 목적: 현재 프로젝트에서 남아 있는 보안 기술부채를 `기능 우선 원칙` 아래에서 명시적으로 관리한다.
- 기준 시점: 2026-03-15 현재 코드 기준
- 원칙: 이미 해결된 항목은 부채 목록에서 분리하고, 아직 남은 항목만 추적한다.

## 현재 상태 요약

### 이번에 해소된 항목
- 비공개 게시글 본문 이미지/파일 접근 제어
  - `/api/files/{id}/view`는 이제 파일 귀속과 접근 제어를 반영한다.
  - 보호 파일은 `file view ticket`과 짧은 TTL의 presigned GET을 거친다.
- SSE access token query parameter 허용
  - 알림 실시간 연결은 `SSE ticket` 방식으로 전환됐다.
  - `JwtAuthFilter`의 query access token 특례는 제거됐다.

### 아직 남아 있는 기술부채
- 공개 GET API 확장 시 서비스 레벨 인가 누락 가능성
- 운영 환경 보안 설정 검증 자동화/체크리스트 부족
- 감사 추적 범위 부족

## 현재 보안 기반에서 잘 되어 있는 점
- API 인증은 JWT 기반 무상태 구조로 운영한다.
- OAuth2 플로우와 일반 API 인증 체인을 분리했다.
- Refresh Token은 HttpOnly Cookie로만 저장한다.
- Refresh/Logout에는 Origin allowlist 검사를 적용한다.
- 게시판/게시글 조회 및 작성은 서비스 레벨 접근 정책으로 한 번 더 검증한다.
- 파일 조회는 공개 파일과 보호 파일 경로를 분리했다.
- 알림 실시간 연결은 SSE ticket 기반으로 분리했다.

## 현재 기준 기술부채

### 1. 공개 GET API 확대 시 서비스 레벨 인가 누락 가능성
- 현재 우려:
  - 일부 GET API는 Security에서 공개 허용하고, 실제 접근 판단은 서비스 레벨 정책에 의존한다.
- 위험:
  - 새 API 추가 시 서비스 레벨 인가 검사를 빠뜨리면 민감 리소스가 공개 노출될 수 있다.
- 현재 결정:
  - 구조는 유지하되, 보안 가이드와 코드 리뷰 기준으로 관리한다.
- 후속 방향:
  - 민감 리소스 패턴별 보안 가이드 유지
  - Controller/Service 체크리스트 보강
  - 중요 조회 API 테스트 보강

### 2. 운영 환경 보안 설정 검증
- 현재 우려:
  - 쿠키 `secure`, 오브젝트 스토리지 `publicBaseUrl`, presigned TTL, 프록시 설정은 운영값에 크게 의존한다.
- 위험:
  - 코드가 안전해도 설정 실수로 보안 수준이 떨어질 수 있다.
- 현재 결정:
  - MVP 단계에서는 기능 우선이지만, 운영 전 점검 체계는 반드시 남긴다.
- 후속 방향:
  - 배포 체크리스트 강화
  - 운영값 점검 자동화 후보 검토

### 3. 비공개/조직형 서비스 수준의 감사 추적 부족
- 현재 우려:
  - 관리자 감사 로그는 일부 있으나, visibility 변경, 권한 변경, 민감 파일 접근까지 충분히 추적하는 수준은 아니다.
- 위험:
  - 업무형 커뮤니티 방향으로 갈 경우 열람/변경 이력 설명력이 부족하다.
- 현재 결정:
  - MVP에서는 후순위
- 후속 방향:
  - 중요 행위에 대한 감사 로그 범위 확장

## 우선순위

### P1
- 운영 환경 보안 설정 검증

### P2
- 공개 GET API 인가 누락 방지 가이드/테스트 보강

### P3
- 감사 로그 확장

## 지금 당장 수정하지 않는 이유
- 현재 단계에서 가장 중요한 것은 제품 핵심 경험과 MVP 완성도다.
- 남은 보안 항목은 중요하지만, 전면 보안 리팩터링으로 기능 개발을 멈출 단계는 아니다.
- 대신 이미 위험도가 높았던 `파일 접근 제어`, `SSE query token`은 우선 해소했고, 남은 항목만 명시적으로 추적한다.

## 재점검 트리거
- 비공개 게시판/비공개 문서 사용 비중이 커질 때
- 업무형 커뮤니티 포지션을 공식화할 때
- 관리자 백오피스 또는 대량 임포트 기능을 더 확장할 때
- 외부 사용자 대상 공개 배포 범위를 넓힐 때
- 팀/조직 단위 고객에게 보여줄 데모 또는 운영 환경을 준비할 때

## 출시 전 최소 점검 체크리스트
- `SECURITY_COOKIE_SECURE` 운영값 확인
- Refresh/Logout Origin allowlist 값 확인
- 오브젝트 스토리지 `publicBaseUrl` 사용 여부 확인
- 보호 파일 `OBJECT_STORAGE_PROTECTED_VIEW_EXPIRE_SECONDS` 값 확인
- 관리자 권한 API가 실제로 서버에서 차단되는지 확인
- 비공개 게시판/비공개 게시글 접근 시나리오 수동 점검

## 후속 작업 후보
1. 공개 GET API 보안 가이드/체크리스트 문서화 보강
2. 운영 환경 보안 설정 점검 문서 및 QA 시나리오 보강
3. 문서/게시판 visibility 변경 감사 로그 추가
4. 민감 파일 접근 감사 로그 범위 확장

## 결론
- 현재 보안은 초기 MVP 단계의 “위험한 핵심 구간”은 상당 부분 정리된 상태다.
- 특히 `파일 접근 제어`, `SSE query token 제거`는 더 이상 MVP 기술부채의 최우선 항목이 아니다.
- 이제 남은 기술부채는 `운영 설정 검증`, `인가 누락 방지 체계`, `감사 추적 강화` 쪽으로 이동했다.
