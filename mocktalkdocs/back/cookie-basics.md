# 쿠키 기본 개념과 프로젝트 적용

이 문서는 쿠키의 기본 동작을 설명하고, 현재 프로젝트에서 왜 쿠키가 두 개로 보이는지까지 정리합니다.

## 쿠키란?

쿠키는 브라우저가 서버로부터 받아 저장하는 작은 텍스트 값입니다.
이 값은 **같은 도메인 + 경로(Path) 조건**을 만족하는 요청에 자동으로 포함됩니다.
즉, 브라우저가 “어떤 요청에 어떤 쿠키를 붙일지”를 규칙으로 관리합니다.

## 쿠키의 주요 속성

- `HttpOnly`: 자바스크립트에서 쿠키를 읽을 수 없게 합니다.
  이 프로젝트는 Refresh Token을 HttpOnly 쿠키로 저장하여 XSS 위험을 줄입니다.
- `Secure`: HTTPS에서만 쿠키를 전송합니다.
  개발 환경에서는 보통 `false`, 운영에서는 `true`로 사용합니다.
- `SameSite=Lax`: 기본적인 CSRF 보호 옵션입니다.
  Lax는 “일반적인 외부 POST”를 막고, 사용자가 직접 주소 이동하는 경우는 허용합니다.
- `Path`: 쿠키가 어떤 경로 요청에만 붙을지를 제한합니다.
  예: `Path=/api/auth/refresh`이면 그 경로로 시작하는 요청에만 쿠키가 전송됩니다.

## Path가 중요한 이유

Refresh Token은 민감한 값입니다.
그래서 **필요한 엔드포인트에만 쿠키가 붙도록** Path를 좁히면 보안성이 올라갑니다.
하지만 Path를 너무 좁히면, 다른 엔드포인트(예: 로그아웃)에서는 쿠키가 아예 전달되지 않습니다.

## 우리 프로젝트는 왜 쿠키가 2개인가?

현재 설정은 아래와 같습니다.

- `refresh_token` 쿠키 (Path: `/api/auth/refresh`)
- `refresh_token` 쿠키 (Path: `/api/auth/logout`)

즉, **이름은 같지만 Path가 다른 쿠키 2개**를 운영합니다.
이렇게 하면:

- `/api/auth/refresh` 호출 시: refresh 전용 쿠키만 전송
- `/api/auth/logout` 호출 시: logout 전용 쿠키만 전송

결과적으로 **리프레시와 로그아웃 모두 정상 동작**하면서, 다른 API에는 쿠키가 붙지 않습니다.

## 실제 흐름

1) 로그인
- 서버가 `Set-Cookie`를 2개 내려줍니다. (refresh/logout 각각)

2) 토큰 재발급
- 브라우저는 `/api/auth/refresh` 요청에만 refresh 쿠키를 붙입니다.

3) 로그아웃
- 브라우저는 `/api/auth/logout` 요청에만 logout 쿠키를 붙입니다.

## 관련 설정 요약

- 쿠키 보안 옵션: `SECURITY_COOKIE_SECURE`
- Origin allowlist(Refresh/Logout만 적용): `SECURITY_ORIGIN_ALLOWLIST`
- Refresh 쿠키 경로: `/api/auth/refresh`
- Logout 쿠키 경로: `/api/auth/logout`

## 주의사항

- 쿠키는 요청 헤더로 자동 전송되므로, Path 설정이 잘못되면 토큰이 전달되지 않습니다.
- 쿠키는 대략 4KB 제한이 있으므로 값이 너무 길어지지 않아야 합니다.
- Access Token은 쿠키가 아니라 Authorization 헤더로 전달됩니다.
- Access Token은 프론트 메모리 저장이라 JS에서 읽을 수 있습니다.
- Refresh Token은 HttpOnly 쿠키라 JS에서 읽을 수 없습니다.

## 쿠키 vs 세션

쿠키는 **클라이언트(브라우저)에 저장되는 값**이고, 세션은 **서버에 저장되는 상태**입니다.
전통적인 세션 인증에서는 서버가 세션을 만들고, 클라이언트는 세션 ID만 쿠키로 보관합니다.

이 프로젝트는 기본적으로 **JWT 기반 무상태(Stateless) 방식**을 사용합니다.
즉, 서버는 세션을 유지하지 않고 Access Token을 검증해서 요청을 처리합니다.
다만 OAuth2 로그인처럼 인증 플로우가 필요한 구간에서는 세션을 **일시적으로 허용**합니다.

정리하면:
- 쿠키: 브라우저에 저장, 서버로 자동 전송
- 세션: 서버에 저장, 보통 쿠키(세션 ID)로 연결
- 현재 정책: API는 Stateless, OAuth2는 필요 구간만 세션 허용

## SameSite 옵션별 차이

SameSite는 쿠키가 **어떤 출처 요청에 포함될지**를 제어하는 옵션입니다.

- `Strict`: 다른 사이트에서 들어오는 요청에는 쿠키가 거의 포함되지 않습니다.
  보안은 가장 강하지만, 외부 링크 이동이나 OAuth 흐름에서 문제가 생길 수 있습니다.
- `Lax`: 일반적인 외부 POST 요청에는 쿠키가 포함되지 않지만,
  사용자가 링크를 클릭해서 이동하는 경우에는 쿠키가 포함될 수 있습니다.
  실무에서 가장 무난한 기본값으로 많이 사용됩니다.
- `None`: 모든 크로스사이트 요청에 쿠키를 포함합니다.
  이 경우 반드시 `Secure`가 필요하며, CSRF 방어를 별도로 강하게 가져가야 합니다.

현재 프로젝트는 **`SameSite=Lax`**를 사용합니다.
이는 Refresh/Logout 같은 민감 요청을 최소한의 CSRF 보호와 함께 운용하려는 선택입니다.

## OAuth2 플로우와 SameSite 충돌 사례

OAuth2 로그인은 보통 아래 흐름을 가집니다.

1) 사용자가 “Google 로그인” 클릭  
2) 브라우저가 Google 인증 화면으로 이동  
3) 인증 후 Google이 우리 서버의 콜백 URL로 리다이렉트  

이때 쿠키가 정상적으로 붙지 않으면 다음 문제가 발생할 수 있습니다.

### 사례 1: SameSite=Strict로 설정한 경우

Strict는 **다른 사이트에서 넘어오는 요청에는 쿠키를 거의 붙이지 않습니다.**
OAuth2 콜백은 “외부 사이트에서 우리 서버로 넘어오는 요청”이기 때문에,
서버가 인증 상태를 확인할 쿠키를 못 받아 **로그인이 실패하거나 state 검증에 실패**할 수 있습니다.

### 사례 2: SameSite=None + Secure 미설정

`SameSite=None`은 모든 크로스사이트 요청에 쿠키를 붙입니다.
하지만 브라우저는 **Secure가 없으면 None 쿠키를 거부**합니다.
즉, 로컬 개발에서 HTTPS를 쓰지 않으면 쿠키가 저장되지 않아 로그인 흐름이 깨질 수 있습니다.

### 사례 3: hidden iframe 기반 로그인 연동

외부 서비스가 iframe을 통해 OAuth2 연동을 시도하면,
브라우저는 iframe 요청을 서드파티로 판단합니다.
이때 Lax/Strict는 쿠키 전송을 제한할 수 있어 로그인/리프레시가 실패할 수 있습니다.

### 왜 우리는 Lax를 쓰는가

Lax는 “사용자가 직접 이동하는 Top-level navigation”에는 쿠키를 허용합니다.
OAuth2 리다이렉트는 일반적으로 브라우저 전체 이동이기 때문에,
Lax로도 대부분 정상 동작합니다.

즉, Lax는 **보안과 호환성 사이의 균형점**으로 선택한 값입니다.
